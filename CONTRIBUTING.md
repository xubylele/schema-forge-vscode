# Contributing to Schema Forge VS Code

Thanks for contributing to `schema-forge-vscode`.

## Branch Policy

All contributors must target the `develop` branch.

- Create feature branches from `develop`
- Open pull requests into `develop`
- Do not open contributor PRs directly to `pre-release` or `master`

## Development Setup

1. Clone the repository.
2. Run `npm install`.
3. Run `npm run watch`.
4. Press F5 in VS Code to launch the Extension Development Host.

## Pull Requests

Use the repository pull request template and include a clear summary, implementation details, and validation notes.
Set the base branch to `develop`.

## Changesets Workflow

This repository uses Changesets for versioning and changelog generation.

### Purpose

Automate versioning, changelog generation, and release packaging for the Schema Forge VS Code extension.

This workflow ensures consistent semantic versioning and a clean release history.

1. Create a feature branch.
2. Run `npx changeset`.
3. Select the affected package and release type (`patch`, `minor`, or `major`).
4. Write a short, clear release note.
5. Commit the generated `.changeset/*.md` file with your feature work.
6. Merge feature work into `develop`.

### Branch Channels

- `feature/*`: contributor branches
- `develop`: integration branch for all contributor changes
- `pre-release`: prerelease channel (`X.Y.Z-next.N`)
- `master`: stable production channel (`X.Y.Z`)

### Pre-release Publish

Run these commands on `pre-release` during prerelease cycles:

1. `npx changeset pre enter next`
2. `npx changeset version`

Push the version commit to `pre-release`. CI publishes with `vsce publish --pre-release`.

`pre-release` should be updated from `develop` by maintainers.

### Promote to Stable

Run these commands on `pre-release` before promoting:

1. `npx changeset pre exit`
2. `npx changeset version`

Open and merge `pre-release -> master`. CI publishes stable with `vsce publish` from `master`.

## Release Model

- Version management is driven by Changesets.
- VSIX artifacts are generated by CI for release branches.
- `develop` is the required target for contributor pull requests.
- `pre-release` publishes Marketplace pre-release versions.
- `master` publishes Marketplace stable versions.

Summary flow: Feature Branch → Changeset → Merge to `develop` → Promote to `pre-release` → Pre-release publish → Promote to `master` → Stable publish.

For CLI release specifics, refer to the `schema-forge` repository documentation.
